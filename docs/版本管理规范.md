# 版本管理规范

> 用户不自行写代码；版本管理由 AI 负责。  
> 配套文件：根目录 [`VERSION`](../VERSION) · [`版本发布日志.md`](./版本发布日志.md)

---

## 文件说明

| 文件 | 作用 |
|------|------|
| `VERSION` | 当前线上版本号（如 `1.0.0`） |
| `docs/版本发布日志.md` | 每次正式发布后的变更记录 |

---

## 发布流程（AI 必须按顺序执行）

1. 完成网站修改，本地预览确认
2. `git commit`（说明本次变更）
3. `git push origin main`，等待 GitHub Actions 部署成功
4. 验证线上可访问：https://cgd1093702532-star.github.io/personal-site/
5. **仅在线上确认成功后**：
   - 按 SemVer 更新 `VERSION`
   - 在 `docs/版本发布日志.md` 顶部追加新版本条目
   - 打 Git 标签 `vX.Y.Z` 并推送
   - 提交并推送版本文件与日志

---

## 版本号规则

| 变更类型 | 版本段 | 示例 |
|----------|--------|------|
| 文案/样式小改、Bug 修复 | PATCH +1 | 1.0.0 → 1.0.1 |
| 新区块、明显功能升级 | MINOR +1 | 1.0.0 → 1.1.0 |
| 大改版、架构重构 | MAJOR +1 | 1.0.0 → 2.0.0 |

---

## 用户常用指令

- **「推送到 GitHub」** → 执行完整发布流程
- **「发布 v1.0.1」** → 使用指定版本号
- **「查看版本日志」** → 打开 `docs/版本发布日志.md`

---

## 禁止事项

- 未验证线上可访问就更新发布日志
- 跳过 `VERSION` 或发布日志直接打 tag
