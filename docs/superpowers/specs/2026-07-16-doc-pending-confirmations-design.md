# 需求文档待确认事项设计

## 目标

每次修改 `docs/miniprogram/pages/*.md` 后，检查该文档当前需求正文的内部一致性。发现逻辑矛盾、描述不一致或必要逻辑遗漏时，在文档最底部以待办形式记录，等待用户明确。

## 判断优先级

1. 正文当前需求
2. 补充说明
3. 变更记录与历史记录

历史记录与当前正文不同不视为矛盾；只检查当前需求正文内部的问题。

## 待办格式

```markdown
## 待确认事项

- [ ] 矛盾：§3.2 与 §3.5 对同一操作描述不同……
- [ ] 不一致：……
- [ ] 遗漏：已描述入口，但未说明失败状态……
```

每条须说明问题类型、涉及章节和需要用户明确的具体问题。没有待确认事项时不保留空章节。
底部待办不使用会标红的 `【矛盾】` 等标记；已有底部红色内容从 Markdown 真源删除。

## 处理规则

- 不替用户决定待确认事项的答案。
- 不根据待办或需求文档反向修改原型。
- 用户明确补充后，将结论写入对应正文。
- 只有条目被勾选为 `[x]` 时才视为已处理；后续整理文档时删除该条。
- 未勾选条目必须保留，不得因重复编辑而丢失。
- 不使用自动扫描脚本；由 AI 进行语义检查。

## 实现

新增一条针对 `docs/miniprogram/pages/*.md` 的 Cursor 项目规则，与现有“原型 → 文档单向”和“只补充不覆盖”规则共同生效。
